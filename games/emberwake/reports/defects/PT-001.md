# PT-001 — Zero-count inventory entries persist after in-battle item consumption

- **Persona:** rusher
- **Seed:** 1101
- **Script:** `reports/playtest/rusher/actions.json`
- **Trace:** `reports/playtest/rusher/trace.jsonl`, lines 249-254 (frame 409+); first observable at line 249
- **Severity:** medium (UI noise at M0; gold-dupe exploit via shop once combined with `shop.gd`)
- **Owning skill:** m8-engine-smith (`src/engine/battle.gd`); secondary: m8-scene-smith (`src/scenes/shop.gd`)

## Repro
Run the rusher script with seed 1101. In battle 4 (single rime whelp at F1 (14,3)), the
party consumes its last `item.warm_draught` via the battle Item command (trace line 249,
battle_event `use.warm_draught`, turn 7). From the next line onward, `state.inventory`
is `{"item.warm_draught": 0}` — the key remains at count 0 instead of being removed.

## Trace excerpt (line 249, trimmed)
```
{"detail":{"ability":"use.warm_draught","effect_op":"heal","result":35,...,"turn":7},
 "event":"battle_event","frame":409,"scene":"battle_menu",
 "state":{...,"inventory":{"item.warm_draught":0},...}}
```
The 0-count entry persists in every subsequent snapshot until the item is repurchased
(line 713+), and would persist into saves (`Game.snapshot()` copies inventory verbatim).

## Expected
`Game.add_item()` erases keys at count <= 0, so inventory never contains 0-count
entries. Items consumed in battle should behave identically.

## Observed
`engine/battle.gd` `_do_party_action` ("item" branch, ~line 249) and `pay_costs`
("item" cost, ~line 335) decrement `inventory[iid]` **directly**, bypassing
`Game.add_item`'s zero-erase. Consequences:
1. Snapshots/saves carry phantom `{"id": 0}` entries.
2. `scenes/inventory.gd` `_rebuild_list` iterates all inventory keys → shows "Warm Draught x0".
3. `scenes/shop.gd` `_refresh` lists the 0-count stack as sellable, and `_sell_input`
   ignores `remove_item`'s false return while still crediting `gold += price/2` —
   selling a phantom stack yields infinite gold (not exercised in this trace; code path
   confirmed by reading `shop.gd` lines 113-121).

## Suggested fix
Route battle item decrements through the same zero-erase path (or erase on <=0 in
battle.gd), and make `shop.gd` `_sell_input` check `remove_item()`'s return value.

## Resolution
**Fixed by m8-engine-smith.** `engine/battle.gd` now routes both the Item command and
`item` ability costs through `_consume_item()`, which erases the key at count <= 0
(mirrors `Game.add_item` semantics); `scenes/shop.gd` `_sell_input` now checks
`remove_item()`'s return and never credits gold for a phantom stack. Regression test
`test_item_consumption_zero_erases` added; suite 430/430 green.

**Verified in the repaired rusher run** (seed 1101): zero lines in
`reports/playtest/rusher/trace.jsonl` contain a <=0 inventory count (previously 460+
lines carried `{"item.warm_draught": 0}` from trace line 249 / battle 4 onward).
Side effect handled: the zero-erase reorders inventory keys after re-acquisition,
which shifted two item-menu cursor positions in the rusher script (battle 8 round 1:
emberdew index 1 -> 0; pre-boss inventory menu: emberdew index 1 -> 2, now behind
ember_key). Script repaired; all 226 battle events replay identically to the pre-fix
run and the run still passes (ending + credits + quit, deterministic rerun
byte-identical, party L5 entering the boss).
